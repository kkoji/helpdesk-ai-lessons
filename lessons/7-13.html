セクション7 AWSへの展開：実運用化

コスト管理とクリーンアップ

このセクションで作成したAWSリソースの費用の目安と、後片付け（リソース削除）の手順をまとめた参照用ページです。

ご自身のAWSアカウントで進めた方は、学習が終わったリソースを放置すると、少額とはいえ課金が続きます。動作確認が済んだら、このページの手順でクリーンアップしてください（UdemyのAWSラボ環境で進めた方は、費用がかからず自動でリセットされるため、この作業は不要です）。もう一度作りたくなったら、deploy.sh を実行すればいつでも数分で再構築できます。

このページの内容

  1. 費用の目安
  2. クリーンアップ手順（A：個別削除 ／ B：destroy.sh）
  3. 削除できたかの確認
  4. OpenAI 側の後片付け
  5. 日頃のコスト管理のヒント

費用の目安

⚠️ 料金は変わることがあります。 以下は執筆時点の目安・考え方です。正確な単価は AWS・OpenAI の公式料金ページで確認してください。

OpenAI API（このコースの主な費用）

項目使う場面目安
gpt-4o-mini（チャット）質問への回答・エージェントの判断多くの場合1質問あたり1円未満
text-embedding-3-smallインデックス構築（1回）＋質問のベクトル化多くの場合1回あたり1円未満

実際の金額は、文書量やプロンプトの長さ、その時点の料金設定によって変わります。とはいえこのコース程度のデータ量であれば、ひととおり試しても数ドル以内に収まるケースがほとんどです。

AWS（サーバーレス中心なので、多くは「使った分だけ」）

サービス課金の考え方このコースの規模での目安
Lambda実行回数 × 実行時間数十回の呼び出しならほぼ0円
API Gatewayリクエスト数同上
S3保存容量＋リクエスト数MBの保存なのでごく少額
DynamoDBオンデマンド（読み書き回数）同上
CloudWatch Logsログの保存容量少額（ただし削除忘れに注意）

ただし、S3やCloudWatch Logsは「保存しているだけ」でも容量に応じて課金されます。放置して課金が続きやすいのは、このS3の保存分とCloudWatch Logsのログです。だからこそ、以下のクリーンアップが大事です。

このコースで作成するリソースは、すべて us-east-1（バージニア北部）リージョンに作られます。以下の削除コマンドも同じリージョンが対象です。
💡 Udemy AWSラボ環境の方は、このページのAWSリソース削除は不要です。 ラボ環境のAWS利用料はUdemy側が負担するため受講者に費用は請求されず、環境はセッション終了時に自動でリセットされます。そのため、下記のクリーンアップを行う必要はありません。
ご自身のAWSアカウントで進めた方は、必ず削除を実施してください。 なお、OpenAI APIはラボ・自アカウントのどちらでもご自身のキーを使うため、そちらの利用料は環境にかかわらず発生します。

クリーンアップ手順

このコースでは2通りの作り方をしたので、削除も2通りに分かれます。どちらで作ったか分からない場合は、A・Bの両方を実行しても安全です（存在しないリソースの削除はスキップされます）。各コードブロックは右上の「コピー」ボタンでコピーできます。

A. コマンドで個別に構築したリソースの削除

このセクションの各レクチャーで、コマンドを一つずつ実行して構築したリソースは、以下のコマンドでまとめて削除できます（deploy.sh で一括構築した場合はBへ）。CloudShellでの実行を推奨します（bash構文を含むため、WindowsのPowerShellでは動きません）。

# 変数（構築時と同じもの）
export AWS_REGION=us-east-1
export ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
export S3_BUCKET=helpdesk-ai-${ACCOUNT_ID}-${AWS_REGION}

# ① API Gateway（チャットボットを公開したレクチャーで作成）
API_ID=$(aws apigatewayv2 get-apis \
  --query "Items[?Name=='helpdesk-lab09-api'].ApiId | [0]" --output text)
[ -n "$API_ID" ] && [ "$API_ID" != "None" ] && aws apigatewayv2 delete-api --api-id ${API_ID}

# ② Lambda関数（各レクチャーで作成した3つ）
for FN in helpdesk-lab07 helpdesk-lab08b helpdesk-lab09; do
  aws lambda delete-function --function-name ${FN} 2>/dev/null && echo "削除: ${FN}"
done

# ③ CloudWatch Logs（関数を消しても残るので明示的に削除）
for FN in helpdesk-lab07 helpdesk-lab08b helpdesk-lab09; do
  aws logs delete-log-group --log-group-name /aws/lambda/${FN} 2>/dev/null
done

# ④ Lambda Layer（全バージョンを削除）
for V in $(aws lambda list-layer-versions --layer-name helpdesk-ai-deps \
  --query "LayerVersions[].Version" --output text); do
  aws lambda delete-layer-version --layer-name helpdesk-ai-deps --version-number ${V}
done

# ⑤ DynamoDBテーブル（会話履歴のレクチャーで作成）
aws dynamodb delete-table --table-name helpdesk-sessions 2>/dev/null

# ⑥ IAMロール（インラインポリシー→管理ポリシーの順に外してから削除）
aws iam delete-role-policy --role-name helpdesk-ai-lambda-role \
  --policy-name helpdesk-ai-inline 2>/dev/null
aws iam detach-role-policy --role-name helpdesk-ai-lambda-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole 2>/dev/null
aws iam delete-role --role-name helpdesk-ai-lambda-role 2>/dev/null

# ⑦ S3バケット（中身ごと削除。取り消せないので注意！）
aws s3 rb s3://${S3_BUCKET} --force
⚠️ ⑦のバケット削除は、中身（文書・インデックス・Layer zip）ごと消え、取り消せません。 もう一度作る場合は、インデックスを構築したレクチャーの手順（または deploy.sh）で再構築できます。

B. deploy.sh で一括構築した場合の削除（destroy.sh）

deploy.sh で構築した環境（関数名 helpdesk-ai-handler、API名 helpdesk-ai-api などの正式名）は、リポジトリ付属の削除スクリプト destroy.sh で一括削除できます（どちらも helpdesk-ai リポジトリの aws/ にあります）。

cd helpdesk-ai/aws
./destroy.sh          # 削除対象を表示し、確認プロンプト（yes入力）の後に削除
FORCE=1 ./destroy.sh  # 確認なしで即削除

destroy.sh は作成と逆順（API Gateway → Lambda → ログ → Layer → IAM → DynamoDB → S3）で確実に削除します。

💡 AとBの両方を実行しても安全です（存在しないリソースの削除はスキップされます）。「どちらで作ったか覚えていない」場合は、両方流してください。

削除できたかの確認

それぞれ「空」または「見つからない」と表示されれば、クリーンアップ完了です。

# それぞれ「空」または「見つからない」ならクリーンアップ完了
aws lambda list-functions --query "Functions[?starts_with(FunctionName, 'helpdesk')].FunctionName"
aws apigatewayv2 get-apis --query "Items[?starts_with(Name, 'helpdesk')].Name"
aws dynamodb list-tables --query "TableNames[?starts_with(@, 'helpdesk')]"
aws s3 ls | grep helpdesk || echo "S3: なし"
aws iam get-role --role-name helpdesk-ai-lambda-role 2>&1 | head -1

OpenAI 側の後片付け

  • APIキーの無効化：学習専用に発行したキーは、使い終わったら platform.openai.com のAPI keys画面 で削除（Revoke）しておくと安心です。
  • 残高の確認：Billing画面でクレジット残高と使用量を確認できます。自動チャージ（オートリチャージ）がオフになっていることも再確認しましょう。

日頃のコスト管理のヒント

  • AWS Budgets：「月◯ドルを超えたらメール通知」という予算アラートを設定できます。少数のBudgetsであれば通常は無料枠内で利用できます。個人アカウントで学習を続けるなら、まず最初に設定しておくのがおすすめです。
  • 請求ダッシュボード：AWSコンソールの「Billing and Cost Management」で、サービス別の利用額をいつでも確認できます。
  • OpenAI Usage：OpenAIプラットフォームの Usage 画面で、日別・モデル別の利用量を確認できます。

再構築したくなったら

すべて削除しても、環境は数分で再現できます。

cd helpdesk-ai/aws
# APIキーの入力待ちになったら、ご自身のキーを貼り付けて Enter（画面には表示されない）
# deploy.sh が S3→インデックス→DynamoDB→IAM→Layer→Lambda→API Gateway を一括構築
read -s -p "OpenAI APIキーを貼り付けて Enter: " OPENAI_API_KEY; echo; export OPENAI_API_KEY; ./deploy.sh

「作っては壊し、また作れる」——これがコード化されたインフラの強みです。安心して片付けてください。
