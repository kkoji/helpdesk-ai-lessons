セクション7 AWSへの展開：実運用化

【ラボ10 後編】会話履歴付きで動作確認 ｜ 実行コマンド一覧

手順①〜⑦はAWS CloudShellで、手順⑧だけご自身のPCのターミナルで実行します。

使い方：各手順の右上にある「コピー」ボタンを押して、CloudShellに貼り付けて実行します。複数行を貼り付けると「複数行のテキストの安全な貼り付け」という確認画面が出るので、内容を確認して「貼り付け」を押してください。
環境ごとの進め方の違い
・UdemyのAWSラボ環境の方：ラボ環境はレクチャーごとにリセットされ、前のラボで作ったリソースは残っていません。このページのブロックを上から順にすべて実行してください（「スキップ可」の手順も実行が必要です）。
・ご自身のAWSアカウントの方：前のラボまでのリソースが残っているため、「スキップ可」の手順は飛ばせます。ただし変数設定を含む手順（土台の準備・URL取得）は毎回実行してください（変数はターミナルのセッションを閉じると消えるため）。

1土台の準備（これまでのラボと同じ手順。変数設定を含むため毎回実行してください。作成済みのリソースはそのまま再利用されます）毎回実行

土台の準備（変数・バケット・ロール・Layer）。作成済みのリソースがあればそのまま再利用されます。最後にAPIキーの入力待ちになるので、ご自身のキーを貼り付けて Enter を押してください（画面には表示されません）。

# ソースコードを取得（取得済みの場合はエラーが表示されますが、そのまま進んでOK）
cd ~
git clone https://github.com/kkoji/helpdesk-ai.git
cd ~/helpdesk-ai

# これまでのラボと同じ変数（セッションが切れていたら再設定）
export AWS_REGION=us-east-1
export ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
export S3_BUCKET=helpdesk-ai-${ACCOUNT_ID}-${AWS_REGION}
export LAMBDA_ROLE_NAME=helpdesk-ai-lambda-role
export LAYER_NAME=helpdesk-ai-deps
# 作業用のS3バケットを用意（作成済みなら既存を使う）
aws s3 mb s3://${S3_BUCKET} --region ${AWS_REGION} 2>/dev/null || echo "既存バケットを使用: ${S3_BUCKET}"

# Lambdaの実行ロールを用意（作成済みなら既存を使う）
aws iam create-role \
  --role-name ${LAMBDA_ROLE_NAME} \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }' 2>/dev/null || echo "既存ロールを使用: ${LAMBDA_ROLE_NAME}"

aws iam attach-role-policy \
  --role-name ${LAMBDA_ROLE_NAME} \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

export LAMBDA_ROLE_ARN=arn:aws:iam::${ACCOUNT_ID}:role/${LAMBDA_ROLE_NAME}

# Lambda Layer（依存ライブラリ）をS3経由で登録
aws s3 cp s3://udemy-helpdesk-public/layers/layer-v1.zip layer.zip --no-sign-request
aws s3 cp layer.zip s3://${S3_BUCKET}/layers/layer-v1.zip
export LAYER_VERSION_ARN=$(aws lambda publish-layer-version \
  --layer-name ${LAYER_NAME} \
  --content S3Bucket=${S3_BUCKET},S3Key=layers/layer-v1.zip \
  --compatible-runtimes python3.11 \
  --query LayerVersionArn --output text)

echo ${LAYER_VERSION_ARN}

# OpenAI APIキーの設定（キーを貼り付けて Enter。画面には表示されない）
read -s -p "OpenAI APIキーを貼り付けて Enter: " OPENAI_API_KEY; echo; export OPENAI_API_KEY

3完成版チャットボットを再構築スキップ可（環境が残っている場合）

完成版チャットボット（テーブル・権限・完成版Lambda・API Gateway）を作り直します。環境が残っている場合はスキップできます（実行しても、作成済みのリソースはそのまま再利用されます）。

# 会話履歴のテーブル＋TTL
aws dynamodb create-table \
  --table-name helpdesk-sessions \
  --attribute-definitions AttributeName=session_id,AttributeType=S \
  --key-schema AttributeName=session_id,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST 2>/dev/null || echo "既存テーブルを使用: helpdesk-sessions"
aws dynamodb wait table-exists --table-name helpdesk-sessions
aws dynamodb update-time-to-live \
  --table-name helpdesk-sessions \
  --time-to-live-specification "Enabled=true, AttributeName=ttl" 2>/dev/null || echo "TTLは設定済み"

# ロールの権限を「S3読み取り＋DynamoDB読み書き」の完全版に
aws iam put-role-policy \
  --role-name ${LAMBDA_ROLE_NAME} \
  --policy-name helpdesk-ai-inline \
  --policy-document "{
    \"Version\": \"2012-10-17\",
    \"Statement\": [
      {
        \"Effect\": \"Allow\",
        \"Action\": [\"s3:GetObject\"],
        \"Resource\": \"arn:aws:s3:::${S3_BUCKET}/*\"
      },
      {
        \"Effect\": \"Allow\",
        \"Action\": [\"dynamodb:GetItem\", \"dynamodb:PutItem\"],
        \"Resource\": \"arn:aws:dynamodb:${AWS_REGION}:${ACCOUNT_ID}:table/helpdesk-sessions\"
      }
    ]
  }"

# 完成版Lambdaをデプロイ
cd ~/helpdesk-ai/aws
zip function-final.zip lambda_function.py
aws lambda create-function \
  --function-name helpdesk-lab09 \
  --runtime python3.11 \
  --role ${LAMBDA_ROLE_ARN} \
  --handler lambda_function.lambda_handler \
  --zip-file fileb://function-final.zip \
  --layers ${LAYER_VERSION_ARN} \
  --timeout 60 --memory-size 512 \
  --environment "Variables={
    OPENAI_API_KEY=${OPENAI_API_KEY},
    CHAT_MODEL_ID=gpt-4o-mini,
    EMBED_MODEL_ID=text-embedding-3-small,
    S3_BUCKET=${S3_BUCKET},
    DOCS_PREFIX=docs/,
    METADATA_KEY=metadata.json,
    INDEX_KEY=vector_index/index.faiss,
    CHUNKS_KEY=vector_index/chunks.json,
    SESSION_HISTORY_TABLE=helpdesk-sessions
  }" 2>/dev/null || echo "既存の関数を使用: helpdesk-lab09"
cd ~/helpdesk-ai

# API Gateway（作成＋呼び出し許可＋ルート2本＋CORS）
export LAMBDA_ARN=$(aws lambda get-function --function-name helpdesk-lab09 \
  --query Configuration.FunctionArn --output text)

# APIは既存があれば再利用（無いときだけ作成）
export API_ID=$(aws apigatewayv2 get-apis \
  --query "Items[?Name=='helpdesk-lab09-api'].ApiId | [0]" --output text)
if [ -z "${API_ID}" ] || [ "${API_ID}" = "None" ]; then
  export API_ID=$(aws apigatewayv2 create-api \
    --name helpdesk-lab09-api \
    --protocol-type HTTP \
    --target ${LAMBDA_ARN} \
    --query ApiId --output text)
fi

aws lambda add-permission \
  --function-name helpdesk-lab09 \
  --statement-id apigateway-invoke \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn "arn:aws:execute-api:${AWS_REGION}:${ACCOUNT_ID}:${API_ID}/*/*" 2>/dev/null || echo "呼び出し許可は設定済み"

export INTEGRATION_ID=$(aws apigatewayv2 get-integrations --api-id ${API_ID} \
  --query "Items[0].IntegrationId" --output text)

aws apigatewayv2 create-route --api-id ${API_ID} \
  --route-key "POST /chat" --target integrations/${INTEGRATION_ID} 2>/dev/null || echo "ルートは作成済み: POST /chat"
aws apigatewayv2 create-route --api-id ${API_ID} \
  --route-key "OPTIONS /chat" --target integrations/${INTEGRATION_ID} 2>/dev/null || echo "ルートは作成済み: OPTIONS /chat"

aws apigatewayv2 update-api --api-id ${API_ID} \
  --cors-configuration "AllowOrigins=*,AllowMethods=POST,OPTIONS,AllowHeaders=Content-Type"

export API_ID=$(aws apigatewayv2 get-apis \

5マルチターン——同じsession_idで2連投して、会話がつながるか確認

マルチターンの確認。同じ session_id で2回質問し、2問目の「それって？」が文脈を踏まえて解釈されることを確認します。①と②は1つずつコピーして実行し、そのつど結果を確認してください（まとめて実行すると出力が続いて読みにくくなります）。

① 1問目：有給休暇の日数をたずねます。

curl -X POST ${API_ENDPOINT}/chat -H 'Content-Type: application/json' \
  -d '{"session_id": "demo-100", "message": "有給休暇は何日もらえますか？"}' -w "\n"

② 2問目：同じ session_id で「それって申請は必要ですか？」と聞きます。

curl -X POST ${API_ENDPOINT}/chat -H 'Content-Type: application/json' \
  -d '{"session_id": "demo-100", "message": "それって申請は必要ですか？"}' -w "\n"

6セッションの分離——別のsession_idで、いきなり「それって？」と聞くと…

セッション分離の確認。別の session_id では文脈が引き継がれないことを確認します。

curl -X POST ${API_ENDPOINT}/chat -H 'Content-Type: application/json' \
  -d '{"session_id": "demo-200", "message": "それって申請は必要ですか？"}' -w "\n"

73つのツールの使い分け

ツールの使い分けの確認。①と②は1つずつコピーして実行し、そのつど結果を確認してください。

① 日付の計算が要る質問（文書の検索 → 日付の計算の連携）。

curl -X POST ${API_ENDPOINT}/chat -H 'Content-Type: application/json' \
  -d '{"session_id": "demo-100", "message": "経費精算の申請締切まで、あと何日ですか？"}' -w "\n"

② 社内文書に無い質問（担当部署へのエスカレーション）。

curl -X POST ${API_ENDPOINT}/chat -H 'Content-Type: application/json' \
  -d '{"session_id": "demo-100", "message": "社内VPNに接続できないエラーが出ています。どうすればいいですか？"}' -w "\n"
